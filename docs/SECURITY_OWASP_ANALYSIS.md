# Анализ приложения по принципам OWASP

Документ фиксирует результаты анализа безопасности в соответствии с [OWASP Top 10:2025](https://owasp.org/Top10/2025/), [Cheat Sheet Series](https://cheatsheetseries.owasp.org/) и правилом проекта [.cursor/rules/security-owasp.mdc](.cursor/rules/security-owasp.mdc). Ниже — находки и рекомендации по устранению.

## A01:2025 Broken Access Control

### 1. API без принудительной аутентификации

**Суть:** Все маршруты REST API доступны без проверки личности вызывающего. Зависимость `optional_api_key` (X-API-Key) реализована в [api/dependencies.py](api/dependencies.py), но **ни один роутер её не использует**.

**Риск:** Любой клиент может вызывать любой endpoint: менять имена игроков, подтверждать/отклонять матчи от имени любого участника, запрашивать уведомления, читать данные по произвольным ID.

**Рекомендации:**

- Включить проверку на уровне приложения:
  - Либо глобально: `app.dependency_overrides` или общий middleware, требующий `X-API-Key` для всех маршрутов, кроме `/`, `/health`, `/docs`, `/openapi-supabase.yaml`, `/docs-supabase`.
  - Либо явно добавить `Depends(optional_api_key)` во все роутеры (players, seasons, divisions, matches), оставив публичными только health и docs.
- Документировать в README: при деплое обязательно задавать `API_KEY` и передавать его с фронтенда/бота в заголовке `X-API-Key` при обращении к API.

### 2. IDOR: доверие идентификаторам из запроса без проверки владельца

**Статус (ветка fix/idor-access-control):** Введён заголовок `X-Player-Id` для идентификации вызывающего. На перечисленных endpoint'ах добавлены проверки владельца/участника; отказы в доступе логируются (OWASP §7).

| Endpoint | Проблема | Рекомендация |
|---------|----------|--------------|
| `PATCH /players/{player_id}` | Обновление имени по `player_id` без проверки, что вызывающий — этот игрок (например, по Telegram/session). | Ввести аутентификацию (API key + идентификатор игрока/сессии) и проверять: вызывающий может менять только своего игрока (например, привязка player_id к telegram_id или к сессии). |
| `GET /matches/pending?player_id=...` | Возврат ожидающих подтверждения матчей по переданному `player_id` без проверки личности. | Требовать аутентификацию и возвращать pending только для того `player_id`, который привязан к текущему вызывающему. |
| `GET /matches/{match_id}` | Любой может получить данные матча по ID. | Если матчи считаются конфиденциальными до подтверждения — ограничить доступ участникам (проверять, что вызывающий — player1 или player2). Иначе оставить как есть, но зафиксировать в политике. |
| `POST /matches/submit-for-confirmation` | В теле передаётся `submitted_by` (player_id). Сервер не проверяет, что запрос действительно от этого игрока. | После ввода аутентификации: извлекать текущего игрока из токена/сессии и сверять с `submitted_by`; отклонять запрос при несовпадении. |
| `POST /matches/{match_id}/notify-pending` | Вызов без проверки участника. Любой может инициировать уведомление по любому `match_id`. | Проверять, что вызывающий — участник матча (player1 или player2), и только тогда проксировать запрос боту. |
| `POST /matches/{match_id}/confirm` и `reject` | Участник матча и роль (соперник/не соперник) проверяются по полям тела запроса (`confirmed_by_player_id` / `rejected_by_player_id`). Личность вызывающего не проверяется. | После аутентификации: привязывать текущего пользователя к `player_id` (например, через telegram_id → player) и требовать, чтобы `confirmed_by_player_id`/`rejected_by_player_id` совпадал с текущим игроком. |
| `GET /players?telegram_id=...` | Возврат игрока по `telegram_id` без ограничений. | Ограничить: возвращать только текущего аутентифицированного игрока по его telegram_id, либо убрать возможность запроса по произвольному telegram_id (если не нужно для админки с отдельной авторизацией). |

### 3. Отсутствие CORS

**Суть:** В [api/main.py](api/main.py) не настроен `CORSMiddleware`. Поведение зависит от ASGI-сервера и окружения.

**Рекомендация:** Явно добавить CORS в FastAPI: разрешать только доверенные origin’ы (например, домен фронта и при необходимости Telegram WebApp origin). Ориентир: [OWASP Cheat Sheet — CORS](https://cheatsheetseries.owasp.org/).

```python
# Пример (подставить нужные origins)
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(CORSMiddleware, allow_origins=["https://your-front.com"], allow_methods=["GET","POST","PATCH"], allow_headers=["*"])
```

### 4. RLS в Supabase: политики «allow all»

**Суть:** В [database/schema.sql](database/schema.sql) и миграциях (003–005, 010) RLS включён, но политики заданы как `USING (true)` / `WITH CHECK (true)` для основных таблиц (`players`, `matches`, `division_players`, `rating_history` и т.д.). Фронт использует anon-ключ, поэтому с точки зрения Supabase любой анонимный клиент может читать/писать все строки в рамках этих политик.

**Риск:** Если кто-то получит anon-ключ (он неизбежно в клиенте), возможен полный доступ к данным в объёме, разрешённом RLS. Сейчас это по сути полный доступ.

**Рекомендации (поэтапно):**

- Определить модель «текущего пользователя» в Supabase: например, JWT с `player_id` или `telegram_id`, выдаваемый вашим бэкендом или кастомной auth-функцией после проверки Telegram initData.
- Переписать RLS так, чтобы:
  - SELECT по `players`: только свой профиль или публичные поля (имя, рейтинг) для остальных.
  - UPDATE `players`: только строка, где `id` = текущий `player_id`.
  - `matches`: чтение/вставка/обновление только для участников (player1_id/player2_id = текущий игрок) или по бизнес-правилам.
  - Аналогично для `division_players`, `rating_history`, `client_sessions` — привязка к текущему пользователю/игроку.

Это большая задача; минимальный первый шаг — включить аутентификацию на API и не полагаться на прямой доступ к Supabase с фронта для мутирующих операций, оставив мутации через ваш API с проверкой владельца.

## A02 / A07: Аутентификация и конфигурация

### 5. Идентификация пользователя только на фронте (Telegram initData)

**Суть:** В [frontend/src/hooks/useTelegram.js](frontend/src/hooks/useTelegram.js) и [frontend/src/hooks/useSupabase.js](frontend/src/hooks/useSupabase.js) используется `webApp.initDataUnsafe?.user?.id` (Telegram user ID). Серверная проверка подписи Telegram Web App initData в коде не обнаружена.

**Риск:** Клиент может подделать или изменить данные, если не проверять подпись на бэкенде. См. [Telegram Web App — validating data](https://core.telegram.org/bots/webapps#validating-data-received-via-the-mini-app).

**Рекомендации:**

- На бэкенде (API или отдельный сервис) реализовать проверку подписи initData (алгоритм из документации Telegram).
- Выдавать после проверки сессионный токен или JWT с привязкой к `telegram_id` (и при необходимости к `player_id`) и использовать этот токен для вызовов API вместо передачи только `player_id`/`telegram_id` в теле/query.
- На фронте по возможности использовать только проверенные данные (после успешного обмена initData на токен).

---

### 6. Секреты и конфигурация

**Суть:** В коде нет хардкода секретов; используются переменные окружения (`API_KEY`, `NOTIFY_SECRET`, `SUPABASE_KEY` и т.д.). Фронт использует только anon-ключ из env — это ожидаемо.

**Рекомендации:**

- Убедиться, что `SUPABASE_KEY` в API — это service_role только в backend, никогда не попадает во фронт и в репозиторий.
- В production задавать `API_KEY` и `NOTIFY_SECRET`, не оставлять пустыми.
- Документировать в README/.env.example необходимость этих переменных и риски при их отсутствии.

## Дополнительно (A09, практики из правила)

### 7. Логирование отказов в доступе

**Рекомендация:** При внедрении проверок доступа логировать отказы (403/401) с минимальным контекстом (endpoint, тип ошибки, без паролей и токенов) и при необходимости настроить алерты при повторных неуспешных попытках. См. [OWASP Cheat Sheet — Logging](https://cheatsheetseries.owasp.org/).

---

### 8. Ограничение частоты запросов (rate limiting)

**Суть:** В проекте не обнаружено rate limiting для API.

**Рекомендация:** Добавить ограничение частоты запросов по IP или по ключу/токену для защиты от брутфорса и злоупотреблений (например, массовая отправка уведомлений через `POST /matches/{id}/notify-pending`). Можно использовать middleware FastAPI или прокси (nginx/cloud).
